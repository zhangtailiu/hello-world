增量同步与补偿闭环设计

目标：实现上游 API 到本地数据库的自动增量同步，支持多实例并发安全、状态可收敛、失败可追溯与人工补偿。

冻结决策： Redis 分布式锁 + fencing token 消费端二次查询（lastId 分页） 部分成功允许推进 checkpoint 失败以人工重放为主 系统失败可归档关闭

背景

  • 上游接口按 id 升序，固定 curPage=1，通过 lastId 向后推进。
  • 每次调用都必须传入 startDate 和 endDate。
  • 生产端先落明细快照，再发送 MQ；消费端按参数回查上游并落业务表，再更新明细状态。
  • 多实例消费存在乱序，不能依赖“最后批次消息”判断最终状态。

约束

  • 同一任务同一时刻仅允许一个生产者活跃；活跃 run 心跳在 1 小时内时跳过新触发。
  • 常规增量起点看 sync_task_checkpoint；若存在超时未完成 run，则优先按 runlog 断点续跑。
  • endBatch 仅提示，不参与终态收敛。
  • 消费幂等键为 (runId, batchNo, sourceId)，实体写入用业务唯一键 upsert。
  • 失败池必须可查询、可重放、可归档且可审计。

状态机

Run 状态机

stateDiagram-v2 [*] --> STARTED STARTED --> RUNNING: "进入生产或消费阶段" RUNNING --> SUCCESS: "produce=PRODUCED 且 pending=0 且 validationFail=0 且 systemFail=0" RUNNING --> PARTIAL_SUCCESS: "produce=PRODUCED 且 pending=0 且 (validationFail>0 或 systemFail>0)" STARTED --> FAILED: "生产阶段不可恢复失败" RUNNING --> FAILED: "接管失败或不可恢复异常"

明细状态机

stateDiagram-v2 [*] --> INIT INIT --> CONSUMING: "批次被消费" CONSUMING --> SUCCESS: "业务入库成功" CONSUMING --> VALIDATION_FAILED: "业务校验失败" CONSUMING --> SYSTEM_FAILED: "技术异常失败" VALIDATION_FAILED --> CONSUMING: "人工重放" SYSTEM_FAILED --> CONSUMING: "人工重放" SYSTEM_FAILED --> ARCHIVED_CLOSED: "人工归档关闭"

终态布尔判定（权威规则）

can_finalize = (produce_status == 'PRODUCED') AND (pending_count == 0)

if can_finalize AND validation_fail_count == 0 AND system_fail_count == 0:
    run_status = 'SUCCESS'

if can_finalize AND (validation_fail_count > 0 OR system_fail_count > 0):
    run_status = 'PARTIAL_SUCCESS'

if produce_status == 'PRODUCE_FAILED' AND unrecoverable == true:
    run_status = 'FAILED'

主流程图

flowchart TD A["调度触发"] --> B["获取 Redis 锁与 fencing token"] B --> C{"存在活跃 run 且心跳小于 1 小时吗"} C -- "是" --> Z["跳过本次触发"] C -- "否" --> D["创建 run_log 并置为 STARTED/RUNNING"] D --> E["构造固定窗口 startDate/endDate"] E --> F["按 lastId 分页抓取，curPage 固定为 1"] F --> G["落 sync_run_batch 与 sync_run_detail 快照"] G --> H["发送 Kafka 消息"] H --> I{"是否还有下一页"} I -- "是" --> F I -- "否" --> J["置 produce_status=PRODUCED 并推进 checkpoint"] J --> K["消费端按 lastId 区间回查并处理"] K --> L["原子更新明细状态与 run 计数"] L --> M["每批次消费完成后执行 tryFinalize(runId)"] M --> N["SUCCESS 或 PARTIAL_SUCCESS 或 FAILED"]

主流程不变式

  • 必须先提交明细快照，再发送 MQ。
  • 生产阶段完成后即可推进 checkpoint（与消费成败解耦）。
  • 每个批次消费完成后都要触发一次 tryFinalize(runId)。

中断续跑规则（runlog 查询快照）

  • 触发时先查该任务是否存在 produce_status in (INIT, PRODUCING) 且 heartbeat_time 超过 1 小时的 run。
  • 若存在：不新建窗口，直接续跑该 run，使用 query_param_snapshot.startDate/endDate 与 resume_last_id 继续拉取。
  • 若不存在：按 checkpoint 创建新 run，resume_mode=NEW。
  • 每生产完一批，原子更新 resume_last_id=last_id_to、resume_batch_no=batch_no、heartbeat_time=NOW()。
  • 仅当 produce_status=PRODUCED 时推进 checkpoint，避免生产中断导致窗口丢段。

触发阶段选路 SQL（示意）

-- 1) 查可续跑 run（超时且未完成生产）
SELECT id, query_param_snapshot, resume_last_id, resume_batch_no
FROM sync_task_run_log
WHERE task_id = #{taskId}
  AND run_status IN ('STARTED','RUNNING')
  AND produce_status IN ('INIT','PRODUCING')
  AND heartbeat_time < DATE_SUB(NOW(), INTERVAL 1 HOUR)
ORDER BY id DESC
LIMIT 1;

-- 2) 命中后接管（fencing token 更新）
UPDATE sync_task_run_log
SET owner_instance = #{instanceId},
    lock_token = #{newLockToken},
    resume_mode = 'RESUME',
    heartbeat_time = NOW(),
    updated_at = NOW()
WHERE id = #{runId};

批次推进断点 SQL（示意）

UPDATE sync_task_run_log
SET resume_last_id = #{lastIdTo},
    resume_batch_no = #{batchNo},
    heartbeat_time = NOW(),
    updated_at = NOW()
WHERE id = #{runId}
  AND lock_token = #{lockToken};

异常流程图

flowchart TD A["收到触发"] --> B{"是否存在 RUNNING 记录"} B -- "是" --> C{"heartbeat 超过 1 小时吗"} C -- "否" --> D["返回 SKIPPED_ACTIVE_RUN"] C -- "是" --> E["创建 RECOVER run 并 fencing 旧生产者"] B -- "否" --> F["进入正常执行"] E --> G["接管未闭合批次或关闭旧 run 为 FAILED"] F --> H["消费阶段出现失败"] H --> I{"失败类型"} I -- "校验失败" --> J["标记 VALIDATION_FAILED，进入失败池"] I -- "系统失败" --> K{"达到上限策略吗"} K -- "否" --> L["标记 SYSTEM_FAILED，等待人工重放"] K -- "是" --> M["人工归档关闭 ARCHIVED_CLOSED"]

失败处理规则

  • VALIDATION_FAILED 默认不自动关闭，需人工重放或人工归档。
  • SYSTEM_FAILED 可人工重放；达到策略上限后可归档关闭。
  • 归档关闭必须落审计表 sync_detail_archive。

补偿流程

  1. 运维或业务人员按任务、状态、失败码、时间范围查询失败池。
  2. 执行重放时，将选中明细置为 CONSUMING，并增加 replay_count。
  3. 重放成功后置 SUCCESS，并回填 entity_id。
  4. 重放失败则更新失败码与失败原因，返回失败池。
  5. 归档关闭时写 sync_detail_archive，并将明细置 ARCHIVED_CLOSED。
  6. 每次批次消费收尾都执行 tryFinalize(runId)，成功更新 1 行即表示终态闭合。

计数更新策略

同一事务内进行：
1) 更新 detail_status
2) 原子增减 run_log 计数（SQL increment/decrement）
3) 必要时更新 batch_status

禁止在应用内先读后写覆盖计数，避免并发丢计数。

消费端即时终态收敛 SQL（推荐）

-- 每个批次消费完成后执行，返回影响行数=1 代表本次成功闭合终态
UPDATE sync_task_run_log
SET run_status = CASE
      WHEN validation_fail_count = 0 AND system_fail_count = 0 THEN 'SUCCESS'
      ELSE 'PARTIAL_SUCCESS'
    END,
    consume_status = CASE
      WHEN validation_fail_count = 0 AND system_fail_count = 0 THEN 'DONE'
      ELSE 'PARTIAL'
    END,
    updated_at = NOW()
WHERE id = #{runId}
  AND produce_status = 'PRODUCED'
  AND finished_batch_count = expected_batch_count
  AND pending_count = 0
  AND run_status IN ('STARTED','RUNNING');

说明：该 SQL 必须在批次状态“首次闭合”后触发，避免重复消费导致的重复计数。

运维指标与告警

核心指标

  • active_run_count
  • run_heartbeat_lag_seconds
  • pending_detail_count
  • validation_fail_rate
  • system_fail_rate
  • archive_close_rate
  • try_finalize_success_count
  • try_finalize_noop_count
  • checkpoint_delay_seconds

告警策略

  • RUNNING 记录心跳超时。
  • 待处理明细持续积压超过阈值。
  • tryFinalize 长时间无成功闭环且 pending 不下降。
  • 校验失败率或系统失败率高于基线。

已确认决策

  • 优先续跑超时未完成 run（按 runlog 快照与断点），否则按 checkpoint 开新窗口。
  • 生产完成后允许推进 checkpoint，消费失败走补偿链路。
  • run 终态由消费端批次收尾时的 tryFinalize 条件更新统一归并。
  • 失败补偿以人工重放为主，支持归档关闭并保留审计。
  • 消费端按 lastId 区间回查，并与已落明细快照做对齐。
